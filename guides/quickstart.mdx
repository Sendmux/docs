This walkthrough takes you from a fresh Sendmux account to a verified sending domain, an active mailbox, and a webhook receiving live delivery events. Plan for about 30 minutes — most of which is waiting for DNS records to propagate.

<Info>
  Before you start, sign in at [app.sendmux.ai](https://app.sendmux.ai) and
  create or join a team. Every step below happens inside that team.
</Info>

## Root vs mailbox API keys

Sendmux has two categories of API key, both prefixed `smx_`:

| Key category          | Base URL                           | Use it for                                                                      |
| --------------------- | ---------------------------------- | ------------------------------------------------------------------------------- |
| **Root** (Management) | `app.sendmux.ai/api/v1`            | Reading and managing providers, domains, mailboxes, webhooks, billing, and logs |
| **Mailbox-scoped**    | `smtp.sendmux.ai/api/v1` (sending) | Sending email and mailbox-scoped endpoints such as `GET /mailbox/me`            |

Use a **mailbox-scoped key** for sending and any `/mailbox/*` endpoint. A root key sent to a mailbox-scoped endpoint is rejected with `403`. The mailbox credential created in step 2 is a mailbox-scoped key.

## What you'll build

By the end of this guide you'll have:

- A verified custom sending domain with the four DNS records (TXT, SPF, DMARC, DKIM) AWS SES requires.
- A live mailbox on that domain with bearer-token credentials.
- A webhook subscription that receives signed delivery events.
- A test email sent and confirmed in delivery logs.

<Steps>
  <Step title="Verify your sending domain">
    Add the domain you want to send from and publish the DNS records Sendmux generates for you.

    **In the dashboard**: open **Mailbox domains** and click **Add domain**. Enter your domain (for example, `mail.acme.com`) and Sendmux returns the records you need to publish at your DNS provider:

    - One **TXT** record proving ownership.
    - One **SPF** record authorising Sendmux to send.
    - One **DMARC** record setting your alignment policy.
    - Three **DKIM** CNAME records issued by AWS SES.

    Publish all six records, then click **Verify** in the dashboard. Verification typically completes within a few minutes once your DNS provider has propagated. The domain status moves from `pending` to `verified`.

    **Or via the API**: see the [Domain management guide](/guides/domain-management) for the equivalent `POST /api/v1/domains` + `POST /api/v1/domains/{id}/verify` flow.

    <Tip>
      You can also use a `@myagent.mx` shared address if you don't have a custom domain yet — skip this step and pick the shared option when creating your mailbox in step 2.
    </Tip>

  </Step>

  <Step title="Create a mailbox">
    A mailbox is a single sending address bound to a verified domain (or to the shared `@myagent.mx` namespace). Each mailbox gets its own credential and can have its own delivery rules.

    **In the dashboard**: open **Mailboxes** and click **Create mailbox**. Pick the verified domain from step 1, choose a local part (for example, `notifications`), and click **Create**.

    The dashboard shows a single one-shot **mailbox credential** (`smx_mbx_...`) that **cannot be retrieved later** — copy it now. The same value authenticates everywhere:

    - As a Bearer token on the HTTP API.
    - As the password for SMTP submission (`mail.sendmux.ai:587`).
    - As the password for IMAP retrieval (`mail.sendmux.ai:993`).

    Store it in your secrets manager. If you lose it, mint a fresh credential from the mailbox detail page — the original value is gone.

  </Step>

  <Step title="Send your first email">
    Use the API key from step 2 to send a test message. Replace `your_key_here` and the email addresses with your own values.

    <CodeGroup>

    ```bash curl
    curl -X POST https://smtp.sendmux.ai/api/v1/emails/send \
      -H "Authorization: Bearer smx_mbx_your_key_here" \
      -H "Content-Type: application/json" \
      -d '{
        "from": { "email": "notifications@your-verified-domain.com", "name": "Your App" },
        "to": { "email": "you@example.com", "name": "Your Name" },
        "subject": "Hello from Sendmux",
        "html_body": "<p>It works!</p>",
        "text_body": "It works!"
      }'
    ```

    ```python Example B
    import requests

    response = requests.post(
        "https://smtp.sendmux.ai/api/v1/emails/send",
        headers={
            "Authorization": "Bearer smx_mbx_your_key_here",
            "Content-Type": "application/json",
        },
        json={
            "from": {"email": "notifications@your-verified-domain.com", "name": "Your App"},
            "to": {"email": "you@example.com", "name": "Your Name"},
            "subject": "Hello from Sendmux",
            "html_body": "<p>It works!</p>",
            "text_body": "It works!",
        },
    )
    print(response.status_code, response.json())
    ```

    ```javascript Node.js
    const response = await fetch("https://smtp.sendmux.ai/api/v1/emails/send", {
      method: "POST",
      headers: {
        Authorization: "Bearer smx_mbx_your_key_here",
        "Content-Type": "application/json",
      },
      body: JSON.stringify({
        from: { email: "notifications@your-verified-domain.com", name: "Your App" },
        to: { email: "you@example.com", name: "Your Name" },
        subject: "Hello from Sendmux",
        html_body: "<p>It works!</p>",
        text_body: "It works!",
      }),
    });
    console.log(response.status, await response.json());
    ```

    </CodeGroup>

    A successful send returns `201 Created` with an `eml_*` message id you can search in your delivery logs. Your send goes through the managed **Sendmux SES** account, which is active by default with a daily limit — see [Sending accounts](/guides/sending-accounts). See the [HTTP sending guide](/guides/sending-via-http) for batch sends, attachments, and idempotency.

    <Tip>
      Prefer SMTP? The same credentials work — see [Send via SMTP](/guides/sending-via-smtp).
    </Tip>

  </Step>

  <Step title="Subscribe a webhook">
    Webhooks let your application react to delivery events (delivered, bounced, complained) and inbound mail in real time.

    **In the dashboard**: open **Webhooks** and click **Add webhook**. Provide:

    - The **URL** that will receive events (must be HTTPS).
    - The **event types** you want — start with `message.bounced` and `message.complained` to learn about deliverability problems.

    On creation, Sendmux returns a signing secret (`whsec_...`) **once**. Copy it into your application's secret store immediately — you cannot retrieve it again. If you lose it, rotate the secret from the webhook detail page.

    Sendmux signs every payload with HMAC-SHA256 over the request body, sent in the `X-Sendmux-Signature` header. Always verify the signature before trusting the payload.

    See [Webhooks setup](/guides/webhooks-setup) for the full event-type catalogue and retry behaviour, and [Verify signatures](/guides/webhooks-verify-signatures) for ready-to-paste verification code.

  </Step>

  <Step title="Verify everything works end-to-end">
    Send another test message — this time to a known good inbox so you can check both delivery and the corresponding webhook.

    1. Send via the API as in step 3.
    2. Confirm the message arrived in the recipient inbox (check spam folder if needed; deliverability improves as your domain reputation builds).
    3. Open **Logs** in the dashboard — your `eml_*` id should appear within seconds with `status: delivered`.
    4. Check your webhook receiver — for a successful delivery, no `message.bounced` or `message.complained` event fires. To exercise the webhook path immediately, send a second message to `bounce@simulator.amazonses.com`; you'll receive a `message.bounced` event within about 30 seconds.

    <Note>
      The bounce simulator address is provided by Amazon SES specifically for testing — it returns a hard bounce without affecting your sender reputation.
    </Note>

  </Step>
</Steps>
